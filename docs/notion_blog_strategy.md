# Notion Blog Strategy

## 목적

이 프로젝트의 글은 단순 개발 일지가 아니라 포트폴리오 문서다. 한 글 안에서 세 독자가 각자 필요한 정보를 얻도록 구성한다.

- 기업 인사담당자: 프로젝트 목적, 기간, 사용 기술, 결과물, GitHub/영상 링크
- 실무 면접관: 환경 설계, RL 문제 정의, 알고리즘 선택 근거, 검증 방법, 실패와 수정
- 개발자 취준생: WSL2 세팅, 의존성 설치, 실행 명령, 디버깅 과정, 재현 절차

## 작성 타이밍

작업 중에는 `docs/process_log.md`에 짧게 기록하고, 아래 조건 중 하나를 만족할 때 Notion 초안으로 확장한다.

- 설계 결정이 바뀌었다.
- 실행 환경 문제가 해결됐다.
- 학습 또는 평가 결과가 나왔다.
- 실패 원인과 해결 방법이 명확해졌다.
- README, 영상, 그래프, HTML 다이어그램처럼 보여줄 산출물이 생겼다.
- GitHub에 의미 있는 단위 작업이 push됐다.

## 추천 글 시리즈

1. 프로젝트 기획편: 왜 분리수거 로봇팔 RL 프로젝트를 선택했는가
2. 환경 구축편: Windows에서 WSL2 Ubuntu 기반 로봇학습 환경 만들기
3. 시뮬레이션 설계편: Fetch-style 테이블탑 GoalEnv를 어떻게 정의했는가
4. 학습 설계편: SAC + HER를 선택한 이유와 알고리즘 확장 구조
5. 실험 결과편: 성공률, 학습 곡선, 영상으로 보는 Curriculum Learning
6. 회고편: Docker, 비전 인식, Sim2Real을 MVP에서 제외한 이유

## 독자별 정보 배치

각 글은 다음 순서를 기본으로 한다.

1. 한 줄 요약: 인사담당자가 빠르게 이해할 수 있는 결과 중심 문장
2. 문제와 목표: 포트폴리오에서 어떤 역량을 보여주려는지 설명
3. 핵심 결정: 면접관이 질문할 만한 설계 선택과 근거
4. 구현/실행 과정: 취준생이 따라 할 수 있는 명령과 파일 구조
5. 결과와 증거: commit, 테스트, 학습 로그, 영상, 그래프
6. 한계와 다음 단계: 과장 없이 남은 리스크와 확장 계획

## 문체 기준

- 기술명은 정확히 쓴다: MuJoCo, Gymnasium, Stable-Baselines3, SAC, HER, WSL2.
- "했다"보다 "왜 그렇게 했는가"를 설명한다.
- 실패와 제약을 숨기지 않는다. 단, 원인/대응/다음 액션까지 함께 쓴다.
- 인사담당자용 문장은 짧게, 면접관용 설명은 근거 중심으로, 취준생용 설명은 명령어 중심으로 쓴다.
