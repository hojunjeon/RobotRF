# Recording Handoff Protocol

이 문서는 개발 에이전트가 기록/포트폴리오 담당 에이전트에게 넘길 정보를 표준화한다.
기록 담당 에이전트는 이 파일과 `docs/recording_handoff_log.md`를 우선 참고한다.

## 역할 분리

- 개발 에이전트: 구현, 실험, 디버깅, 검증을 수행하고 증거를 남긴다.
- 기록 에이전트: 남겨진 증거를 바탕으로 Notion 블로그, README, 포트폴리오 문장, 면접 답변을 구성한다.
- 개발 에이전트는 사용자가 요청하지 않는 한 최종 블로그 글을 작성하지 않는다.

## 프로젝트 기본 정보

- 프로젝트 이름: Robot Sorting RL
- 패키지 이름: `robot-sorting-rl`
- 저장소 주소: `https://github.com/hojunjeon/RobotRF.git`
- 포트폴리오 톤: 채용 포트폴리오형 + 기술 블로그형 + Notion 공개 페이지형
- 핵심 포지셔닝: PM / Agent Operator / RL 실험 설계자 관점

## 기록 패킷 작성 시점

다음 중 하나라도 해당하면 `docs/recording_handoff_log.md`에 항목을 추가한다.

- 프로젝트 범위, 아키텍처, MVP 기준이 바뀜
- 환경 설정, 런타임, 의존성, WSL2/MuJoCo/SB3 관련 결정이 생김
- 시뮬레이션 환경, GoalEnv 계약, reward, curriculum, 알고리즘이 바뀜
- 디버깅에서 원인과 해결책이 확인됨
- 학습/평가/영상 생성이 실행되고 지표나 산출물이 생김
- README, 다이어그램, 영상, GitHub push 등 포트폴리오 증거가 추가됨

## 기록 식별 번호

- 모든 최상위 기록 항목은 `## NNN - YYYY-MM-DD KST - {작업 제목}` 형식을 사용한다.
- `NNN`은 `001`부터 시작하는 3자리 순번이다.
- 새 항목을 추가할 때는 기존 `## NNN -` 제목 중 가장 큰 번호에 1을 더한다.
- 기록 담당 에이전트와 개발 에이전트는 이후 대화에서 `001`, `002`처럼 번호로 항목을 참조한다.

## 기록 패킷 필수 형식

아래 형식을 그대로 사용한다.

```markdown
## NNN - YYYY-MM-DD KST - {작업 제목}

### 오늘 한 일
- 

### 막힌 문제
- 없음

### 해결 방법 / 결정
- 

### 남은 문제
- 

### 증거
- 코드 경로:
- 실행 명령:
- 결과 로그/지표:
- 스크린샷/영상:
- 체크포인트/학습 로그:
- 커밋:

### 기록 담당 에이전트에게 강조할 관점
- 채용 담당자:
- 기술 면접관:
- 개발자/학습자:

### 검증 상태
- 검증 완료:
- 검증 불가:
- 가정:
```

## 증거 규칙

- 파일 경로는 가능한 한 저장소 기준 상대 경로로 쓴다.
- 명령은 실제 실행한 명령만 쓴다.
- 성공률, loss, reward, episode 수 등 숫자는 확인된 값만 쓴다.
- 실패 로그도 남긴다. 실패가 설계 변경의 근거가 될 수 있다.
- 검증하지 못한 내용은 `가정` 또는 `검증 불가`에 분리한다.

## 기록에서 강조할 프레임

- "강화학습을 실행했다"보다 "환경 설계 -> 보상 설계 -> 커리큘럼 설계 -> 실험 검증을 주도했다"를 우선한다.
- AI 도구는 대체자가 아니라 생산성 도구이자 에이전트 운영 사례로 설명한다.
- 단순 구현 기록보다 의사결정, tradeoff, 실패 원인, 다음 실험 계획을 남긴다.
- 채용 담당자용 요약은 짧고 결과 중심으로, 면접관용 설명은 기술 근거 중심으로 쓴다.
